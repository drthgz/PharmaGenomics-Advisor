# Requirements Document

## Introduction

This specification covers the final polish tasks for the PharmaGenomics Advisor project prior to Kaggle hackathon submission. The scope includes: fixing the Dockerfile model-pull process for reliable PID handling, adding docker-compose for single-command deployment, updating the README to reflect the current multi-agent architecture, enhancing the demo script to showcase LLM narratives and agent message-passing, adding detailed inline code comments to key modules, tightening ADK error messages for graceful fallback display, and ensuring all tests pass cleanly. These tasks directly support the evaluation criteria: Technical Implementation (50pts), Documentation (20pts), and The Pitch (30pts).

## Glossary

- **Dockerfile**: The container build specification that packages the project for reproducible execution.
- **Docker_Compose_File**: A `docker-compose.yml` file enabling single-command container orchestration via `docker compose up`.
- **Ollama_Serve_Process**: The background `ollama serve` process started during the Docker build stage to allow model pulling.
- **Model_Pull_Stage**: The Dockerfile RUN step that starts Ollama in the background, waits for readiness, pulls the specified model, and terminates the background process.
- **README**: The project's top-level `readme.md` documentation file.
- **Demo_Script**: The CLI entrypoint `scripts/demo.py` that runs the pipeline and reports results to stdout.
- **AgentMessage**: A structured Pydantic payload exchanged between SupervisorAgent and SpecialistAgents at runtime.
- **LLM_Narrative**: A free-text clinical interpretation paragraph generated by the LLM inference client.
- **ADK_Workflow_Runner**: The module (`adk_workflow.py`) that executes the pipeline through Google ADK's orchestration primitives.
- **Key_Module**: One of the five source files requiring detailed inline comments: `src/agents/supervisor.py`, `src/agents/message_bus.py`, `src/inference/ollama_client.py`, `src/pipeline/orchestrator.py`, `src/security/layer.py`.
- **Test_Suite**: The combined set of unit tests (`tests/unit`) and property-based tests (`tests/properties`) executed via pytest.
- **Inline_Comment**: A Python comment line (prefixed with `#`) placed within a function or class body explaining the purpose, logic, or rationale of adjacent code.

## Requirements

### Requirement 1: Fix Dockerfile Model-Pull PID Handling

**User Story:** As a developer building the Docker image, I want the model-pull stage to track the background Ollama process with `$!` and terminate it cleanly, so that the build does not hang or leave zombie processes.

#### Acceptance Criteria

1. WHEN the Model_Pull_Stage executes, THE Dockerfile SHALL start `ollama serve` as a background process and capture its PID using the `$!` shell variable.
2. WHEN the Ollama_Serve_Process is started during the build, THE Model_Pull_Stage SHALL poll the Ollama HTTP API endpoint at `http://localhost:11434` in a loop with 1-second intervals until it receives a successful response or 30 seconds elapse.
3. IF the Ollama HTTP API does not respond within 30 seconds during the Model_Pull_Stage, THEN THE Dockerfile build SHALL fail with a non-zero exit code and print an error message containing the text "Ollama failed to start".
4. WHEN the Ollama API is confirmed ready, THE Model_Pull_Stage SHALL execute `ollama pull` with the model name specified by the OLLAMA_MODEL build argument.
5. WHEN the model pull completes (successfully or with error), THE Model_Pull_Stage SHALL send SIGTERM to the PID captured via `$!` and call `wait` on that PID to reap the process.
6. THE Model_Pull_Stage SHALL NOT use `/tmp/ollama.pid` or any file-based PID storage mechanism.

### Requirement 2: Add Docker Compose for Single-Command Deployment

**User Story:** As a hackathon reviewer, I want to run `docker compose up` to see the full demo execute, so that deployment is a single command with no manual steps.

#### Acceptance Criteria

1. THE Docker_Compose_File SHALL define a single service named `pharmagenomics` that builds from the project root Dockerfile.
2. THE Docker_Compose_File SHALL map container port 11434 to host port 11434 for Ollama API access.
3. THE Docker_Compose_File SHALL pass the OLLAMA_MODEL build argument with a default value of "medgemma" through the build args section.
4. THE Docker_Compose_File SHALL mount a local `./output` directory to `/app/output` inside the container so that generated reports persist on the host filesystem.
5. THE Docker_Compose_File SHALL be located at the project root as `docker-compose.yml`.
6. WHEN a user executes `docker compose up`, THE Docker_Compose_File SHALL cause the container to build (if not cached) and run the entrypoint producing pipeline output without additional manual intervention.

### Requirement 3: Update README for Current Architecture

**User Story:** As a hackathon reviewer reading the README, I want to see an accurate description of the agent architecture including the LLM inference client, SupervisorAgent, MessageBus, agent-to-agent communication, and property-based tests, so that the documentation matches the actual implementation.

#### Acceptance Criteria

1. THE README SHALL contain a section describing the LLM inference client module and its role in generating clinical narratives via Ollama.
2. THE README SHALL contain a section describing the SupervisorAgent and its role in routing variant analysis tasks to specialist agents.
3. THE README SHALL contain a section describing the MessageBus and its role in dispatching AgentMessage payloads between agents.
4. THE README SHALL describe agent-to-agent communication as a message-passing architecture where the SupervisorAgent delegates to specialist agents (BRCA, EGFR, TP53) via structured AgentMessage objects.
5. THE README SHALL describe the property-based testing approach using the Hypothesis library with a reference to the `tests/properties/` directory.
6. THE README SHALL update the Architecture diagram or description to include the SupervisorAgent, MessageBus, and LLM_Inference_Client components.
7. THE README SHALL update the "Course Concepts Demonstrated" table to reflect the actual agent-to-agent message-passing, LLM inference integration, and property-based test coverage.

### Requirement 4: Enhance Demo Script Output

**User Story:** As a hackathon presenter, I want the demo script to display LLM-generated narratives and agent message-passing events in its output, so that the demo visually showcases the multi-agent intelligence during The Pitch.

#### Acceptance Criteria

1. WHEN the pipeline processes a variant and the LLM_Inference_Client produces a narrative, THE Demo_Script SHALL print the LLM_Narrative text to stdout, prefixed with the gene name and classification.
2. WHEN the SupervisorAgent dispatches an AgentMessage to a specialist agent, THE Demo_Script SHALL log the dispatch event to stdout including the message_type, sender, and recipient fields.
3. WHEN a specialist agent returns a response AgentMessage, THE Demo_Script SHALL log the response receipt to stdout including the message_type, sender, and a summary of the classification result.
4. THE Demo_Script SHALL format all agent communication logs with a visual prefix (e.g., "[Agent]" or "🤖") to distinguish agent events from pipeline summary output.
5. THE Demo_Script SHALL print the pipeline summary statistics (variants analyzed, classifications, drug recommendations, literature bundles, warnings) after all agent communication output.
6. WHEN the pipeline produces a markdown report, THE Demo_Script SHALL print the first 40 lines of the markdown report content to stdout as a preview.

### Requirement 5: Add Detailed Inline Code Comments to Key Modules

**User Story:** As a hackathon reviewer reading the source code, I want to see clear inline comments explaining the purpose and logic of key modules, so that the code quality and documentation score is maximized.

#### Acceptance Criteria

1. THE Key_Module `src/agents/supervisor.py` SHALL contain inline comments explaining: the purpose of the class, the routing logic for dispatching to specialist agents, and the aggregation logic for collecting results.
2. THE Key_Module `src/agents/message_bus.py` SHALL contain inline comments explaining: the purpose of the message bus, the handler registration mechanism, and the message dispatch flow.
3. THE Key_Module `src/inference/ollama_client.py` SHALL contain inline comments explaining: the purpose of the LLM client, the prompt construction logic, the timeout and error handling strategy, and the fallback behavior.
4. THE Key_Module `src/pipeline/orchestrator.py` SHALL contain inline comments explaining: the overall pipeline flow, the purpose of each pipeline stage, and the report assembly logic.
5. THE Key_Module `src/security/layer.py` SHALL contain inline comments explaining: the purpose of the security layer, the PHI detection logic, the injection prevention approach, and the rate limiting mechanism.
6. WHEN inline comments are added, each Key_Module SHALL contain a minimum of 10 inline comment lines within function or method bodies (excluding module-level or class-level docstrings).

### Requirement 6: Tighten ADK Error Messages for Graceful Fallback

**User Story:** As a demo presenter, I want ADK import errors to produce clean, human-friendly messages rather than raw tracebacks, so that a missing ADK package does not derail the demo.

#### Acceptance Criteria

1. IF the `google.adk` package is not installed, THEN THE ADK_Workflow_Runner SHALL raise an ADKNotAvailableError whose message includes the text "ADK runtime not available" and a one-line install hint containing the command `pip install 'google-adk>=2.0.0'`.
2. IF any specific required ADK symbol (Workflow, workflow.START, Runner, InMemorySessionService) is missing from the installed package, THEN THE ADK_Workflow_Runner SHALL raise an ADKNotAvailableError whose message names the missing symbol and states that the installed ADK version may be incompatible.
3. WHEN the Demo_Script catches an ADKNotAvailableError, THE Demo_Script SHALL print a single-line user-friendly message to stderr containing the error text and SHALL NOT print a full Python traceback.
4. WHEN an ADKNotAvailableError is raised, THE Demo_Script SHALL exit with exit code 2 to distinguish from other failure modes.
5. THE ADK_Workflow_Runner SHALL NOT allow any ImportError or AttributeError to propagate uncaught from the ADK integration code path; all such exceptions SHALL be wrapped in ADKNotAvailableError with a descriptive message.

### Requirement 7: Fix Test Suite Import Errors and Failures

**User Story:** As a developer preparing the submission, I want all test files to import correctly and all tests to pass, so that `pytest tests/unit tests/properties -v` exits cleanly.

#### Acceptance Criteria

1. WHEN pytest is invoked with the command `pytest tests/unit tests/properties -v`, THE Test_Suite SHALL collect all test files without raising ModuleNotFoundError, ImportError, or any other collection-time error.
2. THE Test_Suite SHALL have all collected tests pass (zero failures, zero errors) when executed with the project root on the Python path.
3. IF a test file imports a module from `src/`, THE test file SHALL use an import path that resolves correctly when pytest is invoked from the project root directory.
4. THE Test_Suite SHALL produce no import-time warnings related to missing optional dependencies (google-adk import failures are acceptable only if they are caught by try/except at collection time).
5. WHEN the full Test_Suite passes, THE pytest exit code SHALL be 0.

### Requirement 8: Ensure Clean Test Pass

**User Story:** As a developer making the final submission, I want confirmation that the entire test suite (unit + property-based) passes with zero failures, so that the technical implementation score is not penalized.

#### Acceptance Criteria

1. WHEN `pytest tests/unit tests/properties -v` is executed from the project root, THE Test_Suite SHALL exit with code 0 indicating all tests passed.
2. THE Test_Suite SHALL report zero test failures and zero test errors in the final summary line.
3. THE Test_Suite SHALL execute all property-based tests with the configured Hypothesis settings (minimum 100 examples) without any Hypothesis health check failures.
4. IF any test fails due to a code defect (not a test defect), THE source code in `src/` SHALL be corrected to make the test pass rather than modifying or disabling the test.
5. THE Test_Suite SHALL complete execution within 300 seconds (5 minutes) when run without network-dependent tests (excluding tests marked `integration` or `slow`).
