# Requirements Document

## Introduction

This spec addresses the final integration gaps in the PharmaGenomics Advisor multi-agent system prior to Kaggle hackathon submission. The project has all components scaffolded (agents, MCP servers, security layer, VCF parser, ADK workflow) but several pieces are not actually wired together at runtime. These requirements ensure the system genuinely performs AI inference, uses the ADK framework correctly, delegates work between agents with real message-passing, is containerized for reproducibility, and has at least one property-based test exercising core parsing logic.

## Glossary

- **Pipeline**: The end-to-end workflow that takes a VCF file as input and produces a ClinicalReport as output.
- **Ollama**: A local LLM inference server exposing an HTTP API on port 11434.
- **LLM_Inference_Client**: The module responsible for calling `ollama.chat()` to generate clinical narrative text from a specialist agent.
- **ADK_Workflow_Runner**: The module (`adk_workflow.py`) that executes the pipeline through Google ADK's orchestration primitives.
- **Supervisor_Agent**: The top-level agent that receives a variant analysis request and delegates sub-tasks to Specialist_Agents.
- **Specialist_Agent**: A gene-specific agent (BRCA, EGFR, TP53) or function-specific agent (PGX Advisor, Literature RAG) that performs a scoped analysis task.
- **Agent_Message**: A structured payload exchanged between Supervisor_Agent and Specialist_Agents at runtime.
- **VCF_Parser**: The module (`vcf_parser.py`) that transforms VCF file text into structured Variant objects.
- **ACMG_Classifier**: The logic that assigns an ACMG 5-tier pathogenicity classification to a variant.
- **Clinical_Narrative**: A free-text clinical interpretation paragraph generated by the LLM for a classified variant.
- **Dockerfile**: A container build specification that packages the project for reproducible execution.
- **Property_Based_Test**: A test that verifies a property holds across many randomly-generated inputs using the Hypothesis library.

## Requirements

### Requirement 1: Wire Ollama LLM Inference into Specialist Agent

**User Story:** As a hackathon reviewer, I want to see that at least one specialist agent calls a real LLM to produce clinical narrative text, so that the project demonstrates genuine AI-powered interpretation rather than only rule-based logic.

#### Acceptance Criteria

1. WHEN a Variant is classified by the ACMG_Classifier, THE LLM_Inference_Client SHALL call `ollama.chat()` with the model specified in the OLLAMA_MODEL environment variable to generate a Clinical_Narrative for that variant.
2. THE LLM_Inference_Client SHALL include the variant gene, classification, evidence references, and therapeutic relevance in the prompt sent to Ollama, and SHALL NOT send the request if any of these four fields is missing or empty.
3. WHEN Ollama returns a non-empty response of at least 1 character of non-whitespace text, THE LLM_Inference_Client SHALL truncate the generated Clinical_Narrative to a maximum of 2000 characters and attach it to the VariantClassification object.
4. IF Ollama is unreachable, returns an HTTP error, times out, or returns an empty or whitespace-only response, THEN THE LLM_Inference_Client SHALL log a warning at WARNING level and create a placeholder narrative that includes the variant gene name, the ACMG classification value, and the text "LLM-generated narrative unavailable"; both the warning log and placeholder creation must succeed together — if either operation fails, the entire fallback SHALL be considered failed. WHEN Ollama returns HTTP 200 with a completely empty response body (zero bytes), THE LLM_Inference_Client SHALL treat this as a successful response and proceed normally without triggering the fallback path.
5. THE LLM_Inference_Client SHALL enforce a 30-second timeout on the Ollama HTTP call.
6. WHEN the pipeline completes, THE ClinicalReport SHALL include the Clinical_Narrative for each classified variant in the markdown summary output.
7. IF the OLLAMA_MODEL environment variable is not set or is empty, THEN THE LLM_Inference_Client SHALL use "medgemma" as the default model name.

### Requirement 2: Fix ADK API Compatibility

**User Story:** As a developer, I want the ADK workflow code path to actually execute without import errors or API mismatches, so that the project demonstrates correct use of the Google ADK framework.

#### Acceptance Criteria

1. THE ADK_Workflow_Runner SHALL import and use only API symbols that exist in the installed `google-adk>=2.0.0` package, such that invoking the runner does not raise ImportError or AttributeError for any google-adk reference.
2. WHEN the ADK code path is invoked, THE ADK_Workflow_Runner SHALL construct a Workflow object with all five pipeline stages registered as distinct nodes with ordered edges connecting them, and execute the pipeline through ADK's Runner, completing without unhandled exceptions when provided valid input.
3. IF the installed google-adk package does not expose any of the required symbols (Workflow, workflow.START, Runner, or InMemorySessionService), THEN THE ADK_Workflow_Runner SHALL raise an ADKNotAvailableError whose message includes the name of the missing symbol.
4. WHEN the ADK_Workflow_Runner completes execution successfully, THE ADK_Workflow_Runner SHALL return a ClinicalReport object that is a valid instance of the ClinicalReport Pydantic model and contains populated variant_summary, classifications, and markdown_summary fields matching the same data that the deterministic PipelineOrchestrator would produce for the same input.
5. THE ADK_Workflow_Runner SHALL register each of the five pipeline stages (validate/parse, classify, recommend, literature, assemble) as distinct callable node functions within the ADK Workflow edge definition.
6. IF the ADK Workflow run completes but the final event output does not contain a ClinicalReport instance, THEN THE ADK_Workflow_Runner SHALL raise an ADKNotAvailableError with a message indicating that the workflow did not produce the expected report output.

### Requirement 3: Real Agent-to-Agent Communication

**User Story:** As a hackathon reviewer, I want to see the supervisor agent actually delegate tasks to specialist agents with message-passing at runtime, so that the project demonstrates a genuine multi-agent architecture rather than a sequential function pipeline.

#### Acceptance Criteria

1. WHEN the Supervisor_Agent receives a variant analysis request, THE Supervisor_Agent SHALL construct an Agent_Message containing the variant data and route it to the appropriate Specialist_Agent based on the variant gene field matching one of the four supported genes: BRCA1, BRCA2, EGFR, TP53. IF a variant contains a gene not in the supported set, THEN THE Supervisor_Agent SHALL NOT route the message and SHALL skip that variant.
2. THE Specialist_Agent SHALL receive the Agent_Message, perform its gene-specific analysis, and return a response Agent_Message containing the VariantClassification result within a 60-second timeout.
3. WHEN multiple variants target different genes, THE Supervisor_Agent SHALL dispatch Agent_Messages to the corresponding Specialist_Agents concurrently using asyncio and collect all responses before proceeding to aggregation.
4. THE Agent_Message SHALL be a Pydantic model containing: message_type (enum: CLASSIFY_REQUEST, CLASSIFY_RESPONSE, ERROR), sender (str), recipient (str), payload (dict), and timestamp (datetime).
5. WHEN the Supervisor_Agent collects all Specialist_Agent responses, THE Supervisor_Agent SHALL aggregate the VariantClassification results into the classifications list for report assembly, preserving the original variant order.
6. THE Pipeline orchestrator SHALL log each Agent_Message dispatch and response receipt as a structured audit entry containing the message_type, sender, recipient, and timestamp fields.
7. IF a Specialist_Agent is unavailable or times out, THEN THE Supervisor_Agent SHALL log a warning and fall back to the rule-based classification for that variant.

### Requirement 4: Dockerfile for Reproducible Deployment

**User Story:** As a hackathon reviewer, I want to clone the repo and run a single Docker command to see the demo execute end-to-end, so that the project is reproducible without manual environment setup.

#### Acceptance Criteria

1. THE Dockerfile SHALL use a Python 3.10 base image and install all runtime dependencies (excluding dev optional-dependencies) from pyproject.toml.
2. THE Dockerfile SHALL accept OLLAMA_MODEL as a build argument with a default value of "medgemma", install Ollama inside the container, and pull the model specified by the OLLAMA_MODEL build argument during the build stage.
3. THE Dockerfile SHALL expose port 11434 for the Ollama service.
4. WHEN the container starts, THE Dockerfile entrypoint SHALL start the Ollama server, wait until the Ollama API responds on port 11434 within 30 seconds, and then execute the demo script (`scripts/demo.py`) using the default sample VCF file (`data/samples/sample_variants.vcf`). IF any part of the startup sequence fails (Ollama server start, API readiness check, or demo invocation), THEN THE container SHALL fail immediately with a non-zero exit code without attempting subsequent steps.
5. IF the Ollama server does not respond within 30 seconds of container start, THEN THE entrypoint SHALL exit with a non-zero exit code and print an error message indicating the Ollama service failed to start.
6. WHEN the container finishes the demo execution and produces valid output, THE container SHALL print the full content of the generated markdown report file to stdout and exit with code 0, regardless of the demo script's exit code.
7. IF the demo script produces no output and exits with a non-zero exit code, THEN THE container SHALL exit with the same non-zero exit code returned by the demo script.

### Requirement 5: Property-Based Test for VCF Parser

**User Story:** As a developer, I want at least one property-based test using Hypothesis that exercises the VCF parser's round-trip property, so that the project demonstrates rigorous testing methodology.

#### Acceptance Criteria

1. THE Property_Based_Test SHALL use the Hypothesis library to generate random valid VCF record lines via a custom `@composite` strategy that produces Variant objects conforming to the parser's validation rules (allele characters limited to A, T, C, G, N; positions as integers between 1 and 2,147,483,647; quality scores between 0.0 and 99999.0; chromosome strings of 1 to 10 characters; allele strings of 1 to 50 characters).
2. THE Property_Based_Test SHALL verify the round-trip property: for any generated Variant object, calling `format_variant_to_vcf()` then `parse_vcf_line()` on the result SHALL produce a Variant object where all fields (chromosome, position, id, ref_allele, alt_allele, quality, filter_status, info, gene, variant_type, route_status) are equal to the original when compared using Pydantic model equality.
3. THE Property_Based_Test SHALL generate variants with randomized chromosomes, positions, ref alleles, alt alleles, quality scores, and INFO fields including gene annotations, where INFO field keys and values contain no semicolons or equals signs (to avoid parsing ambiguity) and each INFO dict contains between 0 and 10 entries with key lengths of 1 to 20 characters and value lengths of 1 to 50 characters.
4. THE Property_Based_Test SHALL be located in `tests/properties/` and marked with the `@pytest.mark.property` marker.
5. WHEN the test suite runs, THE Property_Based_Test SHALL execute a minimum of 100 examples per property by configuring the Hypothesis settings decorator with `max_examples=100` or greater.
6. IF the round-trip property fails for any generated input, THEN THE Property_Based_Test SHALL report the minimal failing example via Hypothesis shrinking by not suppressing the default shrinking behavior (i.e., `suppress_health_check` and `phases` settings SHALL NOT exclude the shrinking phase).
